Complex Types

Storage design for ARRAY, MAP, and STRUCT types in Mosaic.

Overview

Mosaic supports three complex types: ARRAY, MAP, and STRUCT. Each uses a different expansion strategy that fits Mosaic’s columnar-bucket architecture:

TypeExpansion LevelProjection Granularity
ARRAY Bucket-internal (physical columns within the same bucket) Entire ARRAY column
MAP Bucket-internal (physical columns within the same bucket) Entire MAP column
STRUCT Schema-level (independent top-level columns across buckets) Individual STRUCT fields

ARRAY Type Storage

ARRAY columns use a flattened columnar storage layout. Each ARRAY column is decomposed into physical columns within the same bucket — a lengths column and a values column — both first-class columns that benefit from standard column encoding (DICT, CONST, PLAIN).

Decomposition

An ARRAY<T> column with N rows is stored as:

  1. Lengths column (INT32, N entries): the number of elements in each array. Null arrays are represented by a null in the lengths column (contributing 0 elements). Empty arrays ([]) have length 0 and are non-null.
  2. Values column (type T, M entries): all elements from all rows flattened into a single contiguous column, where M = sum of all non-null lengths. Element-level nulls are tracked by this column’s own null bitmap.

Both columns independently go through the standard encoding selection (DICT, CONST, PLAIN, ALL_NULL), enabling dictionary compression of element values across all arrays in the column.

Example

Input rows:  [1, 2, 3],  null,  [1, 2],  []

Lengths column (INT32):    3, null, 2, 0    ← standard INT32 encoding
Values column  (INT32):    1, 2, 3, 1, 2    ← shared DICT across all arrays

Nested Arrays

ARRAY<ARRAY<INT>> is stored recursively. The outer values column is itself an ARRAY<INT>, which decomposes into its own lengths + values pair. All leaf INT values across all nesting levels end up in a single column, sharing one dictionary. 

ARRAY<ARRAY<INT>> with rows:  [[1,2], [3]],  [[1,2]]

Outer lengths (INT32):   2, 1              ← 2 inner arrays, 1 inner array
Inner lengths (INT32):   2, 1, 2           ← element counts of inner arrays
Leaf values   (INT32):   1, 2, 3, 1, 2     ← all INTs in one column

ARRAY<STRUCT>

When the element type is a STRUCT, each STRUCT field becomes a child column within the same bucket, preceded by a __struct_null__ Boolean column that tracks element-level STRUCT nullability. 

ARRAY<STRUCT<x INT, y UTF8>> physical columns:

Primary:        lengths (INT32, N rows)
Child 0:        __struct_null__ (Boolean, M elements)
Child 1:        x (INT32, M elements)
Child 2:        y (UTF8, M elements)

MAP<K, STRUCT>

MAP values that are STRUCTs expand similarly. The __struct_null__ column and struct field columns follow the key column: 

MAP<UTF8, STRUCT<x INT, y UTF8>> physical columns:

Primary:        lengths (INT32, N rows)
Child 0:        keys (UTF8, M elements)
Child 1:        __struct_null__ (Boolean, M elements)
Child 2:        x (INT32, M elements)
Child 3:        y (UTF8, M elements)

Bucket Internal Format

ARRAY and MAP columns are expanded into physical columns within the same bucket. The bucket header records the child column count and element counts:

Monolithic Bucket

varint   numPrimary             (N logical columns)
varint   numChildren            (C child columns; 0 if no ARRAY/MAP)
repeated C times:
    varint  childElementCount   (total element count for each child)

[encoding flags: 2 bits × (N + C) columns]
[has-nulls flags: 1 bit × (N + C) columns]
[CONST/DICT metadata for all N + C columns]
[null bitmaps: primary=ceil(numRows/8), child=ceil(elementCount/8)]
[column data for all N + C columns]

Paged Bucket

u16     numChildren (LE)
repeated numChildren times:
    u32  childElementCount (LE)
[directory: (N + C) × u32 LE slot sizes]
[column slots: each independently compressed]

Statistics

ARRAY and MAP columns do not support min/max statistics (no meaningful ordering).

MAP Type Storage

MAP columns use the same flattened columnar approach as ARRAY. A MAP<K, V> column is decomposed into three physical columns within the same bucket:

  1. Lengths column (INT32, N entries): the number of key-value pairs in each map.
  2. Keys column (type K, M entries): all keys flattened across all rows.
  3. Values column (type V, M entries): all values flattened across all rows.

All three columns benefit from standard column encoding. Null maps are represented by a null in the lengths column. Empty maps have length 0.

Example

MAP<INT, UTF8> with rows:  {1:"a", 2:"b"},  null,  {3:"c"}

Lengths column (INT32):  2, null, 1
Keys column    (INT32):  1, 2, 3       ← shared DICT across all maps
Values column  (UTF8):   "a", "b", "c" ← shared DICT across all maps

Recursive Nesting

MAP values and keys can themselves be ARRAY or MAP types, which expand recursively. Complex MAP keys (ARRAY or MAP as key type) are rejected at validation. 

MAP<UTF8, ARRAY<INT>> physical columns:

Primary:        lengths (INT32, N rows)         ← MAP lengths
Child 0:        keys (UTF8, M entries)
Child 1:        array_lengths (INT32, M entries) ← inner ARRAY lengths
Child 2:        array_values (INT32, P entries)  ← inner ARRAY leaf values

STRUCT Type Storage

Unlike ARRAY and MAP which expand within a bucket, STRUCT fields are expanded into independent top-level columns that participate in global name sorting and bucket assignment. This maximizes projection pushdown for wide nested schemas — projecting a single STRUCT field reads only the bucket(s) containing that field.

Column ID

Each node in the schema tree is assigned a DFS-incremented column ID (like ORC). IDs are deterministically generated from the schema structure and are not persisted to the file format. STRUCT nodes, their fields, and nested STRUCTs all receive IDs: 

schema: a.b INT, info STRUCT<name UTF8, addr STRUCT<city UTF8, zip INT>>
DFS IDs:
  0: a.b      (INT — column name can contain '.')
  1: info     (STRUCT)
  2: name     (info.name)
  3: addr     (STRUCT)
  4: city     (addr.city)
  5: zip      (addr.zip)

Expanded physical columns use dot-prefixed names (e.g. info.name) for sorting and bucket assignment. Nullable STRUCT levels get a <name>.__null__ column. All internal column matching — projection resolution, STRUCT reassembly, null bitmap lookup — uses column IDs, not names. Column names may be non-unique when a top-level column name coincides with an expanded STRUCT path.

Physical Column Expansion

STRUCT(name UTF8, age INT) as "info" (nullable) expands to:
  info.__null__    (Boolean, STRUCT null bitmap, column_id=1)
  info.age         (INT32, column_id=3)
  info.name        (UTF8, column_id=2)
→ 3 independent columns, sorted by name and distributed across buckets

Nested STRUCT

Nested STRUCT fields expand recursively. Each nullable STRUCT level gets its own __null__ column: 

STRUCT(name UTF8, addr STRUCT(city UTF8, zip INT) NULLABLE) as "info" (nullable):

  info.__null__          (Boolean)   ← info-level null (column_id=1)
  info.addr.__null__     (Boolean)   ← addr-level null (column_id=3)
  info.addr.city         (UTF8, column_id=4)
  info.addr.zip          (INT32, column_id=5)
  info.name              (UTF8, column_id=2)
→ 5 independent columns

Per-level __null__ columns preserve Arrow STRUCT null semantics: addr = null is distinguishable from addr = {city: null, zip: null}.

STRUCT Fields with Complex Types

STRUCT fields can be ARRAY or MAP types. These become independent top-level columns that expand within their own bucket using the standard flattened columnar mechanism: 

STRUCT(id INT, tags ARRAY<INT>) as "info":
  id    → ordinary INT32 column
  tags  → independent ARRAY column, expands in its bucket as lengths + values

Schema Serialization

Top-level STRUCT columns are serialized with type byte 20 and the full field hierarchy in the schema section. On deserialization, the reader expands the STRUCT type into physical leaf columns internally. This is self-describing — no dot-naming heuristic is needed to recover the nesting structure.

Original field order is preserved through serialization: the STRUCT type descriptor records fields in their original order, and the reader reconstructs the StructArray with the same field order after roundtrip.

Projection (Schema-based)

Projection accepts an Arrow Schema subset (like Lance’s ProjectionRequest::Schema). Partial STRUCTs are expressed naturally as Schema subsets — no dot-path parsing needed. 

// Rust
reader.project_schema(&Schema::new(vec![
    Field::new("info", DataType::Struct(Fields::from(vec![
        Field::new("name", DataType::Utf8, true),
    ])), true),
]))?;

# Python
reader.project(pa.schema([pa.field("info", pa.struct([pa.field("name", pa.utf8())]))]))

Supported projection modes:

  • Whole STRUCT: include the full STRUCT field in the projected schema.
  • Leaf projection: include a partial STRUCT with only the desired fields. Ancestor __null__ columns are auto-included.
  • Mixed: any combination of top-level columns and partial STRUCTs, output in the projected schema’s field order.

Projected fields are validated: non-existent fields or type mismatches (e.g. projecting a STRUCT field as a primitive) produce errors.

Column names and STRUCT field names can contain any characters including .. The Schema-based API avoids all dot-path ambiguity. Java retains a project(String[]) convenience for top-level columns (STRUCT leaf projection requires the Schema-based API).

Column Ordering

Physical columns are sorted by (name, column_id) for deterministic bucket assignment. The column_id tie-breaker ensures a total order even when duplicate physical names exist (e.g. top-level "info.age" and STRUCT leaf "info.age").

Statistics

Expanded STRUCT leaf columns support min/max statistics via physical column name (e.g. stats_columns: ["info.age"]). If a stats column name matches multiple physical columns, the writer rejects the ambiguous selection. Stats are collected from expanded physical arrays after STRUCT null propagation.

Why ORC-style over Parquet-style?

  • Simplicity: The lengths + child approach maps naturally to Mosaic’s bucket architecture where each column is independently encoded and compressed. No rep/def level encoding/decoding needed.
  • Common-case efficiency: CONST encoding for lengths achieves near-zero bytes when all arrays have the same length (feature vectors, embeddings). Parquet’s rep/def adds per-value overhead that only pays off at deep nesting (≥ 3 levels).
  • Debuggability: Each physical column is a standard column with its own encoding flags, null bitmap, and data section. No special reconstruction algorithm needed beyond reading lengths and slicing values.

Validation Rules

  • Column names and STRUCT field names may contain any characters.
  • Physical columns are sorted by (name, column_id) for deterministic ordering.
  • STRUCT must have at least one field.
  • MAP key type cannot be ARRAY, MAP, or STRUCT (complex keys rejected).
  • Sorted MAP is not supported.
  • Non-nullable STRUCT fields are validated on write: null values under non-null ancestor rows are rejected.
  • Projection validates field existence and type matching; missing fields or STRUCT/non-STRUCT type mismatches produce errors.
  • Stats column selections must not be ambiguous (reject if multiple physical columns share the same name).
